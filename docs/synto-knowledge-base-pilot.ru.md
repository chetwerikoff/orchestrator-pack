# Пилот базы знаний synto (одна книга)

Методология для **локального пилота на одной книге** с
[synto](https://github.com/kytmanov/synto) — MIT-инструмент «LLM-wiki» с
маршрутизацией по индексу и встроенным stdio MCP-сервером. В репозитории
остаётся только этот документ; vault и wiki живут на твоей машине.

**Спека в очереди:** `docs/issues_drafts/96-synto-knowledge-base-pilot.md`  
**Английская версия:** `docs/synto-knowledge-base-pilot.md`

**Версия synto:** рассчитано на **0.6.x**; запиши `synto --version` в таблицу
результата — между минорными версиями могут меняться MCP-инструменты и поведение.

## Что это (и чем не является)

- **Это:** пошаговый runbook для **тебя (оператора)** и готовые промпты для
  **vault-agent** — отдельной сессии агента **в папке vault**, не AO worker и
  не внутри `orchestrator-pack`.
- **Не это:** интеграция в worker-flow, библиотека из многих книг, коммит
  vault/wiki в git. Следующие шаги — только после вердикта go/no-go внизу.

Вердикт go/no-go оценивает **полезность index-routing synto на одной книге** —
находит ли рабочий вопрос нужную **опубликованную** статью. Это **не** доказательство
hybrid/BM25+embedding или масштаба на много книг.

## Только локально (не коммитить)

Держи всё ниже **вне** git worktree `orchestrator-pack` и **вне git** этого репо:

| Артефакт | Где обычно | Зачем |
|----------|------------|-------|
| Synto vault (wiki, черновики, SQLite) | напр. `/home/you/synto-pilots/my-book-wiki/` | Synto сам коммитит с префиксом `[synto]` — внутри pack засорит историю |
| Файл книги (PDF / экспорт) | на выбор; в vault через `synto add` | Гигиена репо |
| `~/.config/synto/config.toml` | домашняя папка | API-ключи провайдера |
| MCP в клиенте | Cursor `mcp.json`, настройки Claude | Локальный stdio для `synto serve` |

**Изоляция:** состояние в SQLite. Отдельная папка vault со своим
`.synto/state.db` — не шарь одну БД между агентами (в pack известны deadlock'и
на общем SQLite).

**Пути:** везде **абсолютные** (shell, MCP JSON, промпты). На WSL synto и MCP-клиент
в **одной среде** (оба WSL или оба нативный Windows), чтобы был один бинарник
`synto` и один путь к vault.

## Безопасность vault-agent

В папке появятся статьи, сгенерированные LLM; в тексте теоретически может быть
чужая «инструкция». Поэтому:

- **Не AO worker** — не запускай шаги synto через `ao spawn` из этого pack.
- **Минимум инструментов** — файлы/shell только в vault; без git pack, без YAML
  оркестратора, без админки системы.
- **Секреты не в чат** — ключи в `~/.config/synto/config.toml`, не в промпты и
  не в таблицу результата.

### Запрещено vault-agent (на любом шаге)

Не делать без явного решения оператора:

| Запрещено | Почему |
|-----------|--------|
| `run --auto-approve` / массовый approve без ревью | Пропуск ручной проверки |
| `watch` | Фоновые изменения во время ревью или MCP |
| `serve` с HTTP/streamable | Открытый в сеть сервер без авторизации |
| `answer_question` (MCP) или `synto query --synthesize` | Синтез на лету, не проверка «прочитал статью» |

Промпты **по смыслу** (агент сам подбирает флаги synto), кроме списка запретов.

**Перед повтором дорогого шага** (import, ingest/compile, approve): глянь состояние
(`synto doctor`, число draft/published, `git log` с `[synto]`). Без `--force` на
add, без массового re-approve, пока не ясно, что восстанавливаешь после сбоя.

**Одна команда synto за раз** — не параллель ingest/compile/approve/serve.

## Проверка «потребления» (обязательна перед go/no-go)

После **published**-статей:

1. **Рабочий вопрос** — как спросил бы в задаче, не «открой статью X».
2. **Детерминированное чтение** — `find_concept` или `search_articles`, потом
   `read_article`.
3. **Только published** — не draft/verified в `.drafts/`. MCP по умолчанию
   скрывает черновики (`min_status` → `published`).

## Шаги оператора

По порядку. На шагах **Промпт** — сессия агента с cwd = **папка vault** (после
шага 3; до этого — родительская папка будущего vault).

На шагах 5–6 **не** запускай `synto watch` и не держи MCP-сервер включённым.

### 0. Подготовка (только оператор)

1. Одна книга, которую можно обрабатывать локально.
2. **Абсолютный** путь vault **вне** `orchestrator-pack`, напр.
   `/home/you/synto-pilots/<book-slug>-wiki/`.
3. Только **PDF, Markdown или plain text** через `synto add`. EPUB/MOBI/сканы —
   сначала конвертируй сам в PDF/MD.
4. **Объём:** вся книга или глава/выборка. У `textbook` по умолчанию ~25
   концептов, если не переопределить в `synto.toml`. Запишешь объём и лимит в
   таблицу результата.
5. **Лимит трат** (по желанию): потолок по API или по страницам до ingest.
6. Новый чат агента (Cursor и т.п.) с корнем в этой папке. Это **не** AO worker
   для pack.

### 1. Установить synto

**Промпт — вставить vault-agent:**

```
Установи synto (стабильная версия, Python 3.11+) через pip или uv tool.
Проверь `synto --version`. Ничего не ставь в репозиторий orchestrator-pack.
Сообщи версию и способ установки.
```

### 2. Настроить LLM-провайдера

**Промпт — вставить vault-agent:**

```
Запусти интерактивную настройку synto: fast (анализ) и heavy (написание) модели.
Если есть Ollama — предпочти локально; иначе cloud-провайдер оператора.
Ключи только в ~/.config/synto/config.toml, не в vault и не в чат.
Выполни `synto doctor` и отчитайся о связи с обеими ролями моделей.
```

### 3. Создать vault

Подставь `<VAULT_PATH>` из шага 0.

**Промпт — вставить vault-agent:**

```
Создай новый synto vault по пути <VAULT_PATH> (абсолютный путь вне git worktree
orchestrator-pack). Проверь структуру и наличие synto.toml. Дальше все шаги —
из этой папки. Не создавай vault внутри orchestrator-pack и не внутри чужого
git-репозитория — только свежая папка, где git root (если есть) = сам vault.
```

### 4. Импортировать книгу

Подставь `<BOOK_FILE>` и `<SOURCE_TYPE>` (`textbook`, `paper`, `notes` …).

**Промпт — вставить vault-agent:**

```
Перед импортом: объясни, почему <SOURCE_TYPE> подходит этой книге (textbook vs
paper; без типа PDF по умолчанию = paper). Импортируй <BOOK_FILE> с этим типом.
Покажи, что появилось в raw/ и .synto/sources/, и какой source type записан.
```

### 5. Ingest и compile

**Промпт — вставить vault-agent:**

```
Прогони полный pipeline ingest + compile для этого vault: из книги — черновики
концепт-статей. Без auto-publish — черновики в wiki/.drafts/. Без watch.
Сколько черновиков, были ли ошибки compile.
```

### 6. Ревью → published

**Промпт — вставить vault-agent:**

```
Просмотри черновики в wiki/.drafts/ и опубликуй только достаточно точные для
пилота. Интерактивный review/approve synto, не bulk auto-approve. Итог: минимум
одна статья со status: published в wiki/ (не только в .drafts/). Сколько
опубликовано / отклонено / пропущено.
```

### 6b. Лёгкая проверка здоровья wiki

**Промпт — вставить vault-agent:**

```
Выполни `synto doctor`, `synto eval` и `synto maintain --dry-run` для vault.
Отчитайся о структурных проблемах (битые ссылки, frontmatter, низкий confidence,
single-source у published), которые помешают MCP-проверке. Чини только явные
блокеры; concept curation — только если попросит оператор.
```

### 7. Подключить stdio MCP и проверить «потребление»

**Руками оператора:**

1. Добавь synto в MCP-клиент (пример Cursor — **абсолютный** путь vault, та же
   среда, что в шаге 1):

```json
{
  "mcpServers": {
    "synto": {
      "command": "synto",
      "args": ["serve", "--vault", "/absolute/path/to/your-vault"]
    }
  }
}
```

2. Перезапусти клиент. `synto --version` в среде клиента = версии из шага 1.

**Промпт — вставить vault-agent (с MCP):**

```
Через только synto MCP tools ответь на рабочий вопрос по опубликованной wiki:

"<TASK_QUESTION>"

Правила:
- Сначала `find_concept` или `search_articles`, потом `read_article` по лучшему
  совпадению. НЕ вызывай `answer_question` и `synto query`.
- Только статьи со status published (не черновики).
- `trace_lineage` (или аналог) по прочитанной статье — опирается ли выдержка на книгу.
- `concept inspect` по имени концепта — гомонимы / over-merge.
- Верни: концепт, путь/заголовок, status, выдержку, одну строку про lineage.
```

`<TASK_QUESTION>` придумай сам — реальный вопрос по книге (напр. про отказоустойчивость,
консистентность, идемпотентность).

### 8. Стоимость и счётчики

**Промпт — вставить vault-agent:**

```
По локальным метрикам synto для vault (doctor, state DB, логи) отчитайся:
- версия synto (`synto --version`),
- провайдер (local / cloud),
- примерная стоимость или токены за весь conversion,
- число published-статей в wiki/,
- число неопубликованных черновиков.
Без API-ключей в ответе.
```

### 9. Записать результат (только оператор)

Перенеси данные в **Результат пилота** ниже. Один абзац go/no-go про
**полезность index-routing**. Если концепт в wiki есть, а вопрос не попал —
запиши как terminology-recall gap (кандидат на hybrid-search), не как «synto сломан».

### 10. Убрать MCP (только оператор)

После записи результата:

1. Закрой сессии с synto MCP.
2. Удали или отключи `synto` в конфиге MCP-клиента.
3. Перезапусти клиент — сервера synto нет.
4. В таблице: **teardown done: да/нет**.
