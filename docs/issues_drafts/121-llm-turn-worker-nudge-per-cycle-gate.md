# LLM-turn worker nudges must pass the same per-(PR,cycle,intent-class,target) dedup gate as the #332 scripts
GitHub Issue: #384

## Goal

When the orchestrator runtime is an LLM (cursor), the orchestrator turn composes its own worker-bound `ao send` nudges from `orchestratorRules` prose (PING DISCIPLINE, ROUND PROGRESSION, `waiting_worker_review_response`, `merge.ready` handling). These sends must be subject to the **same shared per-(PR, worker-iteration-cycle) settle/dedup gate** that #332 enforces for the deterministic reconcile scripts, so that an episode the worker has already been nudged for — and is already acting on — cannot receive additional redundant nudges within the same cycle, regardless of how many reconcile-state items the turn observes. The outcome: at most one worker nudge per (PR, worker-cycle, intent-class, worker-target) across **all** emitters (scripts + LLM turn) — redundant same-class nudges collapse to one (each caller submits one candidate; the tuple resolves by first-valid-claim, and the delivered payload is the first-claimer's — sufficient because competing same-class nudges carry equivalent actionable content), a distinct-class actionable instruction is not falsely cross-suppressed (it is delivered as its own message), and a materially-new same-class instruction arriving after that class's message is already sent routes to existing escalation rather than a duplicate send.

## Binding surface

This commits the repo to: **every orchestrator→worker nudge `ao send`, whether emitted by a deterministic reconcile script or composed by the LLM orchestrator turn, passes through one shared mechanical gate before delivery via an atomic claim on the full `(PR, cycle, intent-class, worker-target)` key, and a duplicate nudge for an already-served tuple is fail-closed suppressed.**

- **`worker-target` is a stable logical recipient identity bound to an authoritative durable source, not a raw session id.** The target component is the **logical** PR-owning worker plus an explicit **generation**, and the resume-vs-replacement distinction MUST be derived from an **observable durable record — the AO PR-ownership claim**: a new `ao spawn --claim-pr` (a new session claiming the PR) is a **new generation = replacement** (its first nudge is **not** suppressed by the prior generation's record), whereas a worker resuming under the **same** PR-claim lineage (new session id, same claim) is the **same** generation/target (not re-nudged for an already-served tuple). The classifier is the PR-claim record/event, not a fixture-chosen guess — so implementations cannot arbitrarily pick (which would duplicate after resume or starve a replacement). Storage design is the planner's; the **source** of the distinction is fixed. Binds to the repo's worker crash/resume + session-rotation classes (#33/#91 false-idle, replacement-worker).

- **Single-winner across surfaces, via an atomic claim-before-send.** The per-cycle settle/dedup state established by #332 becomes the authoritative arbiter for the LLM-turn nudge path too — not a parallel, script-only store. The send/suppress decision MUST be an **atomic claim** (CAS / cross-process lock with stale-claim recovery) on the full `(PR, cycle, intent-class, worker-target)` key, not a read-then-write check: two callers (e.g. a reconcile script and the LLM-turn wrapper) evaluating the same tuple concurrently must resolve to exactly one winner, never both-observe-empty-both-send. This reuses the atomic single-winner claim precedent of #100/#308 (review-start claim), now for the nudge-send verb. Planner picks the lock/CAS primitive.
- **A sibling's worker-observable nudge participates in the claim — by transport-through where it composes its own message, by claim-around where AO owns the transport.** The single-winner claim owns **worker-observable nudge delivery** for a tuple, whoever the emitter is. Two integration shapes, because AO core is out of scope:
  - **Transport-through** (the pack composes the message and calls `ao send`): goes through the tokenized wrapper like any caller — acquire the token, send, finalize; suppressed if the tuple is already claimed.
  - **Claim-around** (the delivery is **internal to an AO command** the pack cannot wrap — notably `ao review send <run-id>`, which takes only a run id and delivers to the worker itself): the pack **acquires the tuple claim *before* invoking the command and finalizes after** — the claim, not a transport token, is what enforces single-winner (if the tuple is already claimed, the pack does **not** invoke `ao review send` for it). This needs no AO core change and still gives at-most-one worker-observable delivery per tuple.
  What stays **out of scope / not re-gated** is narrower: (a) the sibling's **internal delivery-eligibility logic** (which review run, which CI episode, run-state idempotency), and (b) any **non-worker-observable** channel output. Those proceed under the sibling's own gate. The earlier "counts for dedup but is never gated" framing was contradictory (Codex catch): a worker-observable delivery cannot both count toward single-winner and be ungated — owning the decision-to-deliver **is** the gating. **Target scoping:** because the claim is per `(PR, cycle, intent-class, worker-target)`, a sibling delivery only contends with nudges of the **same** intent-class to the **same** worker-target; a non-worker-observable review-channel write contends with nothing and needs no claim.
- **Single-runtime store access; one canonical store.** All gate callers (the reconcile scripts and the LLM-turn wrapper) execute in the **single configured runtime** (cursor on WSL/Linux pwsh) and access the store from that **one OS** — so the lock/CAS primitive needs only **same-OS cross-process** arbitration, which is what acceptance tests. A genuine Windows-PowerShell-vs-WSL concurrent race against the store is **not** part of the deployment (no second-OS gate caller) and is therefore **not** a binding requirement — avoiding an untestable cross-OS guarantee. Defensively, the gate still **canonicalizes** a path presented as `/mnt/c/...` or `C:\...` to one store identity (reuses #332's "honor custom state path"), so a stray path-form difference cannot create two stores; the adoption check verifies all callers resolve the same canonical store id in the configured runtime. **Resolving to the same path is necessary but not sufficient** — the chosen primitive MUST still arbitrate when two **same-OS** callers race the same tuple (path identity alone can let both win).
- **Dedup is keyed by the full (PR, worker-cycle, intent-class, worker-target) tuple — not bare (PR, cycle).** Worker nudges are not interchangeable: a review-findings "fix & report" nudge, a generic liveness ping, a merge-ready revalidation, a CI-green continue-hand-off, and a findings *delivery* are distinct **intent classes**. The shared accounting record carries a normalized intent class so a *redundant same-class* nudge is suppressed (the incident: 5 same-class review-findings nudges collapse to one) while a *distinct-class* nudge (e.g. a real liveness/hand-off instruction after findings were delivered earlier in the cycle) is **not** falsely cross-suppressed. Planner defines the normalized class set.
- **One intent-class per atomic claim — no distinct-class merging in this issue.** Each (PR, cycle, intent-class, worker-target) gets its **own** independent atomic claim and at-most-one send. Distinct intent-classes are **not cross-suppressed** (so a liveness/hand-off nudge is never swallowed by an earlier findings nudge) and are **not merged into one consolidated message** either — each delivered class is its own worker message. This kills the incident (5 *same-class* review-findings nudges collapse to one via same-class dedup) without building a multi-intent envelope. **Scoped out (follow-up):** merging several distinct-class intents into a single coalesced worker message — that needs an envelope-episode model (immutable covered-class set + all-or-none multi-key finalize) whose complexity is not justified by the reported incident. See the `parked-root-cause` block.
- **Same-class payload selection: first-claim-wins (no arbitration).** Each invocation submits **one** candidate for a tuple, and the mechanism deliberately has no submit/coalesce phase — so there is no point at which the gate observes multiple same-class candidates to compare. The tuple `(PR, cycle, intent-class, worker-target)` therefore resolves by **first valid claim wins**; the delivered payload is the first-claimer's. This is sufficient because the incident's competing same-class nudges carry **equivalent** actionable content (all "you have review findings, fix and report"). A "richest payload wins" arbitration is explicitly **not** required (it would need the parked collection window); the acceptance test asserts only that concurrent same-tuple candidates yield exactly one send.
- **Closed, versioned intent-class taxonomy; unknown classes fail closed.** The normalized intent-class set is a **closed, versioned classifier**, not ad-hoc per-phrase labels — otherwise slightly-different LLM-composed phrasings fall into fresh classes and the same-class storm returns under new labels. The incident surfaces (review-trigger nudge, `waiting_worker_review_response` ping, `merge.ready` revalidate — all "fix & report review findings") MUST normalize to the **same** review-findings class (pinned by fixture). An unrecognized/unclassifiable nudge fails closed by **one** deterministic rule shared by all callers: it maps to a single versioned `unknown-worker-nudge` class whose **cycle boundary is the current PR head SHA** (the one durable datum every caller agrees on from `gh pr view`, bound to the worker-target generation) — so all callers derive the **same** cycle key for the same unclassifiable episode, a head advance is a **new** unknown-class cycle (preventing indefinite suppression), and within a head it is at-most-one-per-`(PR, head-SHA, unknown-class, worker-target)`. It is never minted as a fresh per-phrase bucket (which reopens the storm) and never silently dropped (which starves the worker).
- **Sibling-surface flow is explicit (per integration shape).** A sibling delivering a **worker-observable** nudge MUST **win** the tuple's claim **before** that delivery; if the tuple is already claimed it does **not** deliver (single-winner), while its **non-worker-observable** channel output proceeds under its own gate and needs no claim. A pack-composed sibling send wins a transport **token** and delivers through the wrapper; an AO-internal sibling delivery (`ao review send`) wins the **claim around** the command and only then invokes it. On a win it finalizes the claim atomically. **No-op resolution:** if a sibling wins the claim but its own eligibility then no-ops the delivery (already-delivered / determined not-worker-observable), the claim MUST be deterministically resolved — released or finalized-as-non-delivery — so it neither suppresses a later real LLM nudge for the tuple nor leaves a stale claim for the recovery path to time out.
- **Post-SENT same-class material change never re-sends the tuple.** The already-SENT `(PR, cycle, intent-class, worker-target)` tuple is never re-delivered. The #261 / #281/#373 paths carry only the **original** send's **delivery-uncertainty** (was the *already-issued* message received?) — they do **not** re-send *new* content under the old tuple. A materially-**new** same-class instruction (the prior send genuinely succeeded; the worker is healthy) is therefore **not** delivered on the old tuple at all: it surfaces as **exactly one** observable outcome — a **distinct operator-visible escalation** (audit + notify-class action: "new same-class content for an already-served tuple cannot be delivered without a duplicate"). It is **not** reclassified-and-resent under another intent-class (that would be another worker send) and **not** silently dropped. This keeps both invariants (no duplicate of a served tuple; no silent loss) with one deterministic behavior. It is **not** turned into a new ad-hoc dedup key that would let same-tuple duplicates bypass the gate. The full `(PR, cycle, intent-class, worker-target)` key stays the single discriminator everywhere (post-SENT, fail-closed, matrix, escalation); "same tuple again, already sent" = suppressed (escalation handled elsewhere), not re-sent — but a **different worker-target** in the same PR/cycle/class is a distinct tuple and is **not** suppressed.
- **No cross-process collection window required.** Because there is no distinct-class merge, there is no multi-caller pre-send buffer to own: each surface independently performs the atomic claim for its (PR, cycle, intent-class, worker-target). The incident regression still holds — the review-trigger, `waiting_worker_review_response`, and `merge.ready` nudges in the incident are the **same** review-findings class, so the first claim wins and the rest suppress, regardless of which process evaluates first.
- **Shell-safe transport for LLM-composed payloads.** Worker nudge text is LLM-composed and can contain multiline content, quotes, backticks, `$`, `;`, markdown fences, branch names, and WSL/Windows paths. The adapter MUST pass the payload via stdin / temp-file / structured payload with **no shell interpolation** (the #281/#373 `ao send --file` transport contract), never as interpolated pwsh/WSL command arguments — otherwise messages truncate, malform, or get shell-interpreted. Fixtures cover multiline, quotes, backticks, `$`, `;`, both path forms, and non-ASCII.
- **Claim→send→finalize lifecycle (transaction boundary defined).** The claim has a durable lifecycle so neither ordering failure mode occurs: recording "nudged" only **after** transport success would let a crash-after-send duplicate on resume; recording **before** would let a failed send permanently suppress retry/escalation. Require a claim lifecycle with **two distinct terminal failure outcomes**, not one (names illustrative; planner owns the schema): `CLAIMED → SEND_ATTEMPTED → SENT | FAILED_DEFINITIVE | UNCERTAIN`. A **`FAILED_DEFINITIVE`** outcome (the send provably did **not** dispatch — e.g. a clean pre-dispatch error) is safely **retryable** and does not permanently suppress. An **`UNCERTAIN`** outcome (a crash or timeout **after** `ao send` was dispatched, when delivery may or may not have happened) MUST **not** auto-retry/resend. It does **not** assume a confirm-receipt-then-replay capability the existing arbiter lacks (#281/#373 boundedly escalates an uncorrelated `dispatch_unknown`; it neither persists nor replays the payload): instead an `UNCERTAIN` claim is resolved to a **bounded operator-visible escalation** (the ambiguous-delivery signal) and the tuple stays **served** — it is **never** auto-re-armed or re-derived into a resend (for `review-findings`, a head advance within the run is the *same* tuple, so "re-derive on head advance" would resend it — forbidden). The at-most-once guarantee is kept by never resending the served tuple; if work genuinely remains, it reaches the worker only when a **new cycle boundary** opens (e.g. a new review run = a *different* tuple), not as a re-arm of the uncertain one. Every claim carries a concrete **claim-lease TTL** — a configured default (env-overridable, e.g. tied to the reconcile cadence) **clamped to the AO report-stale value** so a crashed caller can never hold `CLAIMED` indefinitely: past the lease, a stale `CLAIMED` (pre-dispatch) claim is released and retried on the next tick, and a stale `SEND_ATTEMPTED` is treated as `UNCERTAIN` (escalate, never blind resend). The lease bound is what makes first-nudge liveness observable (a fixture asserts a crashed-CLAIMED claim is recovered within the lease). Consistent with the #111/#339 write-ahead-claim lifecycle, the #232/#281/#373 consumed-after-flush delivery contract, and #342 terminal-phase semantics.
- **Cycle identity is per-intent-class and durable, not a per-caller heuristic.** Each intent-class derives its cycle from **its own natural boundary** — there is no single global "cycle" that must be both "same review round" and "new CI transition" at once. Concretely: the **review-findings** class's cycle is the **review round** (the AO review-run id) — a head advance *within* the round (e.g. `ef6e1c7 → a25b5c1`, same review-run, `openFindingCount` falling) is the **same** review-findings cycle; a new review round (new run) is a **new** one. The **ci-green-handoff** class's cycle is the **CI red→green transition on a head** — a new green transition is a **new** ci-green cycle. These never contradict, because dedup is per `(PR, cycle, intent-class, worker-target)`: in the incident scenario (head advances **and** CI goes red→green on it), the review-findings nudge is suppressed *as the same review-findings cycle* while a ci-green-handoff nudge may arm *as a new ci-green cycle* — different intent-classes, different tuples, both deterministic. All surfaces MUST compute each class's cycle key from the **same durable record**, never from inferred "same finding set" / "fresh head" guesses. **Every required intent-class has an observable durable cycle boundary** (exact key fields are the planner's; the boundary is single-sourced and consumed identically by all callers):

  | intent-class | durable cycle boundary |
  |---|---|
  | `review-findings` | the review round (AO review-run id) — head advance within the round = same cycle |
  | `findings-delivery` (`ao review send`) | the same review round (AO review-run id) as its findings |
  | `ci-green-handoff` | the CI red→green transition on a head (per head) |
  | `ci-failure` | the CI red episode key (head + aggregate red-period, per #342) |
  | `liveness`/quiescence ping | current PR head SHA + worker-target generation (head advance = new cycle) |
  | `unknown-worker-nudge` | current PR head SHA + worker-target generation |

  Golden fixtures assert two callers derive the same key for each class.
- **Fail-closed, but liveness-bounded.** If the gate cannot read the shared state, resolve the worker's live cycle state, or determine whether this episode was already nudged, it MUST suppress (no nudge) rather than emit — never a silent unconditional send. Suppression on unresolved state is non-terminal (re-evaluable next turn), but **repeated** unresolved fail-closed suppressions MUST escalate to an operator-visible signal within a **concrete, observable bound** — a configured default of **3 consecutive unresolved evaluations or 15 minutes, whichever comes first** (env-overridable, e.g. `AO_NUDGE_GATE_UNRESOLVED_ESCALATE_*`), with a **hard ceiling equal to the configured AO report-stale value** (the exact value AO uses, not an approximation): escalation MUST fire no later than `report-stale`, and any env override is **clamped to `report-stale`** (a larger value is reduced to it) — a boundary fixture asserts escalation at exactly that derived value — so the gate cannot silently starve the *first* legitimate nudge (replacing a silent-duplicate-send failure with a silent-suppression failure is not acceptable). "Bounded" alone is insufficient — the default and ceiling are part of the contract and are asserted by fixture. The escalation **counter** may be grouped at (PR, cycle) for operator visibility, but grouping is for escalation only — it MUST NOT suppress a distinct `(intent-class, worker-target)` tuple. Intersects #261.
- **No gate-side busy-deferral: rely on AO safe-enqueue + dedup (avoids a TOCTOU).** The gate does **not** try to defer-if-busy. A nudge to a momentarily-busy worker is **safe**: AO enqueues it (proven enqueue-safe behavior — a message to a busy session is queued, not lost), so there is no "do not interrupt" requirement to enforce. The actual harm in the incident — the worker re-polling CI because of repeated "report when CI green" nudges — is prevented by **per-tuple dedup** (one `ci-green-handoff` nudge per CI transition, one `review-findings` nudge per round), **not** by busy-suppression. This deliberately drops a gate-side busy probe: an idle-probe-then-send cannot be made race-free (`ao send --no-wait` disables the busy wait but is **not** a conditional send-if-idle, so a worker that goes busy between probe and dispatch would still receive it) — and since enqueue is safe, the race does not matter. The adapter still uses the **non-blocking** send form (`--no-wait` or equivalent) so a call never blocks the tick ~600s on a busy session; AO enqueues and the worker drains it. Because there is no busy-suppression, there is no stale-busy lease to bound.
- **CI-green rider containment via dedup, not a lease.** A nudge carrying "report `ready_for_review` when CI green" re-triggers the worker's `sleep && check-runs` loop and advances the head into review-outdated **only when it is emitted repeatedly** — which is exactly what per-tuple `ci-green-handoff` dedup prevents (at most one such nudge per CI red→green transition per head/target). No CI-settle lease/heartbeat is needed: the containment is a consequence of the single-winner claim on the `ci-green-handoff` tuple, and a genuinely new transition (new head, new red→green) is a new tuple that legitimately arms once.
- **Does not narrow legitimate escalation.** The first nudge per cycle, the #261 quiescent-worker fallback, and the #281/#373 delivery-failure escalation paths remain reachable — this gate suppresses **duplicate same-cycle** nudges, never a first nudge or a genuine liveness/escalation signal.
- **Structured audit (not a bare token).** Each gate decision records a structured audit row — at minimum: PR number, **logical worker id**, **session-generation**, **raw session id**, **target-resolution reason/source** (how resume-vs-replacement was classified), surface, computed cycle key, intent-class, state-store identity, decision (SEND/SUPPRESS) + reason, claim/delivery phase, and the send target — so stale issue state, wrong cycle key, missing adoption, false suppression of a replacement worker, and fail-closed liveness stalls are all diagnosable. Bare `SEND|SUPPRESS` is insufficient (the project's standing failure class is silent state transitions).

The chosen mechanism (design-analysis gate, cost/risk/sufficiency): **Option B (hardened) — a claim-token + transport-chokepoint adapter.** Each caller (a reconcile script or the LLM turn) classifies its **one** candidate nudge into a normalized intent-class, requests an atomic single-use claim token for `(PR, cycle, intent-class, worker-target)`, and — if granted — sends via file/stdin and finalizes; if not granted (the tuple was already claimed), it suppresses. There is **no** submit/coalesce phase, **no** durable pre-claim queue, and **no** cross-process collection window (those belong to the parked distinct-class envelope). **Failed/crashed delivery is recovered by re-derivation, gated by the claim phase:** because the first nudge for a tuple is re-derivable from live reconcile state, recovery is by re-evaluation on the next reconcile tick — but it MUST honor the claim lifecycle phase so it cannot resend a delivery that may have already happened:
  - a **pre-send `CLAIMED`** claim that left no delivery (caller exited before `ao send`, or released the claim without dispatching) is recovered by the **claim-lease TTL** (stale-claim recovery) and re-derived on the next tick. There is no busy-deferral branch (the gate does not defer-if-busy; AO enqueues to a busy worker).
  - a **`SEND_ATTEMPTED`** (uncertain-delivery) claim — caller exited/timed out *after* `ao send` but before finalize — MUST **not** be auto-released for resend; it routes to the #281/#373 delivery-confirmation/reconciliation path (confirm-then-finalize or escalate), so a possibly-delivered nudge is never blindly resent (preserves crash-resume at-most-once).
This is why no durable deferred-payload queue is needed: eventual first-nudge delivery is a property of reconcile re-derivation for *pre-send* claims, while *post-send* uncertainty is resolved to bounded operator escalation (never a blind resend, and never assuming a confirm-then-replay the arbiter lacks). Raw `ao send` to a worker is denied at the **process boundary for repo-owned call sites in the configured runtime** — the same mechanism and scope #318/#324 use for the review-start verb: pack `scripts/` is prepended to PATH so an ordinary `ao send <worker> …` invocation resolves to a pack shim that **denies** the raw worker-send verb and admits only the tokenized adapter path; the real `ao` is reachable only via the non-turn-visible indirection (`.ao/autonomous-real-binaries.json`, per #318/#324), so the LLM turn and repo scripts cannot reach it by ordinary lookup. This is **not** a claim that a hand-crafted absolute-path call or a deliberately modified `PATH` is blocked — that is the audited operator/break-glass boundary, not an in-scope guarantee (a token check alone cannot stop a call that goes *around* the wrapper, which is exactly why the deny lives at the PATH-resolved process boundary for the controlled callers). On top of the deny, the tokenized wrapper (`journaled-worker-send.ps1` / successor) MUST reject a worker nudge that lacks a valid single-use token bound to `{PR, cycle key, intent-class, worker session/target, phase}`. Together these make a repo-owned script, the LLM turn, or stale config unable to bypass the gate by ordinary invocation while still looking like a normal journaled send. A break-glass manual send is **operator-only** (never reachable from the LLM/runtime nudge paths), MUST carry an explicit `{PR, cycle, intent-class, worker-target, reason}`, and MUST **acquire the same atomic tuple claim BEFORE delivery** (then send, then finalize the accounting record) — exactly like any caller — so a concurrent automated caller cannot claim+deliver the same tuple in the window before the record exists. (Finalizing only *after* sending would leave that race open.) The "non-dedup-counting" variant is rejected because it would reopen the duplicate the claim closes. When the LLM path is not routed through this adapter, the gate is fail-closed per #342 — `ao start` succeeds with the LLM-turn nudge surface disabled (degraded), not a hard `ao start` refusal. Rejected: **Option A** (orchestratorRules prose "one nudge per cycle") — #318 established prose gating of the LLM turn is unreliable; and **Option C** (remove all nudge composition from the LLM turn into scripts) — correct invariant but high migration cost/regression risk on fresh-head scenarios, out of proportion to current harm. B-hardened is the cheapest sufficient executor with acceptable risk given tests + Codex review, and it absorbs GPT-pass-1's atomicity/lifecycle/non-bypassable findings without the full Option-C migration.

### Operator adoption

- The gate is only effective when the LLM-turn worker-send path actually routes through the dedicated nudge command. Adoption MUST be **fail-closed with ONE observable behavior** (per #342, not #318's hard refuse): when the live `orchestratorRules` does not route worker nudges through the gated command, **`ao start` succeeds but the LLM-turn nudge surface is disabled (degraded health)** — the deterministic reconcile scripts keep running, only the ungated LLM nudge path is suppressed until adoption is in place. It does **not** refuse `ao start` outright (that would brick the orchestrator over one surface). The adoption check reports the degraded surface to the operator.
- Non-bypassability is **process-boundary**, not prose: the live `orchestratorRules` exposes an **explicit gated command** as the sanctioned API, **and** pack `scripts/` is on PATH so a bare `ao send <worker>` resolves to the deny-shim (the #318/#324 mechanism). Both are required — the explicit command removes ambiguity about the API, the PATH-resolved deny-shim makes the raw verb unreachable. The adoption check asserts both: the nudge instruction names the gated command **and** the raw worker-send verb is denied at the process boundary for repo-owned call sites in the configured WSL/pwsh runtime. (Scope: ordinary invocation by controlled callers — an absolute-path call to the real binary is the audited operator/break-glass boundary, not an in-scope guarantee.)
- Post-PR operator step: merge the orchestratorRules clause into the live (gitignored) YAML, then `ao stop` / `ao start`, and verify with the adoption check. This is gitignored-config adoption (no tracked-file PR for the YAML itself); the tracked deliverable is the gate command + adoption checker + tests.

1. **Producer-emission — shared settle store:** fixture proves a script-recorded nudge suppresses an LLM-turn nudge for the same `(PR,cycle,intent-class,target)`, and a distinct intent-class is not cross-suppressed.


2. **Producer-emission — transport chokepoint:** fixture proves raw/un-tokened worker sends are rejected and duplicate same-tuple nudges are suppressed before `ao send`.

```producer-emission
producer: orchestrator-pack
datum: orchestrator-worker-send-chokepoint
expected: reject
proof-command: npm test -- llm-nudge-gate-chokepoint
```

- [ ] A nudge already recorded for `(PR, cycle, intent-class, worker-target)` by a **reconcile script** suppresses a subsequent LLM-turn nudge for the **same** tuple — fixture asserts SUPPRESS before `ao send`; a paired negative fixture asserts a **different** intent-class or worker-target in the same cycle is **NOT** suppressed.
- [ ] The reverse holds: an LLM-turn-recorded nudge suppresses a later script nudge for the same tuple — single-winner is symmetric across surfaces (with the same distinct-class/target negative case).
- [ ] **Atomic claim under concurrency:** two callers (script + LLM-turn wrapper) evaluating the same `(PR, cycle, intent-class, worker-target)` **simultaneously** resolve to exactly one SEND and one SUPPRESS — concurrent two-caller fixture proves both-observe-empty-both-send cannot happen; a distinct-tuple concurrent pair both send.
- [ ] **Claim lifecycle / crash-resume (two failure outcomes):** a `FAILED_DEFINITIVE` claim (send provably not dispatched) is retried and does not permanently suppress; an `UNCERTAIN` claim (crash/timeout **after** `ao send` dispatched) is **not** auto-resent — it resolves to a **bounded operator-visible escalation** (no confirm-then-replay) and the tuple stays served (never re-armed/re-derived into a resend); remaining work reaches the worker only via a **new cycle boundary** (a different tuple). A stale `CLAIMED` is released+retried; a stale `SEND_ATTEMPTED` is treated as `UNCERTAIN`. Fixtures for each transition.
- [ ] **Per-class cycle key (golden fixtures):** all callers compute each intent-class's cycle key identically from durable records; golden fixtures cover same-head, head-advance-same-review-round (same review-findings cycle), new review round (new review-findings cycle), red→green transition (new ci-green cycle), and the **both-at-once incident case** (head advances **and** CI red→green) — asserting the review-findings nudge suppresses *and* a ci-green-handoff nudge may arm, as two distinct tuples (no contradiction).
- [ ] **Head-advance-same-cycle:** worker advances the PR head while addressing the same review round (`openFindingCount` falling) → same cycle key → no new nudge armed.
- [ ] **New-cycle:** worker hands off a fresh ready head and a new review round opens (or a new red→green CI transition) → exactly one nudge fires (delivered via non-blocking send; AO enqueues if the worker is momentarily busy). Recovery honors the claim phase: a crashed **pre-send** `CLAIMED` is recovered by the claim-lease TTL and re-derived; a **post-send** `SEND_ATTEMPTED` exit resolves to bounded operator escalation (never auto-resend).
- [ ] **Fail-closed + measurable liveness bound:** with shared state unreadable or worker cycle state unresolved, the gate suppresses and records a non-terminal re-evaluable outcome (never a silent send); a fixture asserts escalation fires **within the configured bound** (default 3 consecutive unresolved evals / 15 min) under persistent unresolved state — not merely "eventually" — and a **boundary fixture asserts an override is clamped to the exact AO `report-stale` value** (a larger override escalates at `report-stale`, not later). The escalation counter may be grouped at (PR, cycle) for visibility but grouping MUST NOT suppress a distinct (intent-class, worker-target) tuple.
- [ ] **No busy-suppression / safe-enqueue:** a fixture asserts the gate does **not** withhold a nudge because the worker is busy — it is delivered (AO enqueues for a busy session) and the CI-poll re-arm storm is prevented by per-tuple `ci-green-handoff` dedup, not by busy-suppression (so there is no stale-busy lease to bound).
- [ ] **Worker-target identity from the PR-claim record:** generation is derived from the AO PR-ownership claim, not a fixture guess — (a) a worker resuming under the **same PR-claim** (new session id) is the **same** target (not re-nudged for an already-served tuple); (b) a **new `--claim-pr`** is a **new** generation/target whose first nudge is **not** suppressed by the prior generation's record; (c) a stale old session still present does not suppress the replacement; fixtures key off the claim record.
- [ ] **Break-glass is claimed:** the override is not reachable from LLM/runtime paths; it requires `{PR, cycle, intent-class, worker-target, reason}` and **acquires the same atomic claim before delivery** (then finalizes). Fixtures: (a) after a break-glass send a later automated send for the same tuple is suppressed; (b) a **concurrent** automated caller racing the same tuple resolves to exactly one send (the break-glass claim blocks the race — no double-send).
- [ ] **Multi-surface storm regression (the incident):** given the opk-rev-689 / PR #380 fixture — 3 findings sent on `ef6e1c7` (`sentFindingCount=3, openFindingCount=0`), worker advanced to `a25b5c1` with CI green — the review-trigger, `waiting_worker_review_response`, and `merge.ready` surfaces evaluating in one turn produce **at most one** worker nudge total, not three-to-five.
- [ ] **CI-green-rider containment (via dedup):** at most one `ci-green-handoff` nudge is delivered per CI red→green transition per head/target — fixture asserts repeated reconcile/LLM evaluations on the same green head do not re-emit the "report ready_for_review when CI green" rider (so the worker's `sleep && check-runs` re-arm storm cannot recur), while a new transition on a new head legitimately arms once.
- [ ] **No legitimate suppression:** the first nudge of a new cycle, the #261 quiescent-worker fallback, and #281/#373 delivery-failure escalation each still fire (fixtures assert they are NOT suppressed).
- [ ] **Sibling worker-observable delivery participates in the claim:** fixture asserts (a) if an LLM nudge already claimed the tuple, the sibling's worker-observable delivery does not fire — for `ao review send` the pack does **not invoke** it (claim-around), for a pack-composed sibling send the token is refused; (b) the sibling's **internal eligibility** and **non-worker-observable** output proceed regardless (not gated); (c) a sibling that wins and then no-ops resolves its claim (no stale claim, no false suppression of a later LLM nudge).
- [ ] **Canonical store identity across runtime path forms:** script callers and the LLM wrapper resolve the **same** canonical state-store id when the repo path is presented as WSL `/mnt/c/...` and as Windows `C:\...` (fixture asserts one store, not two).
- [ ] **Intent-class dedup, not bare cycle:** findings *delivery* earlier in a cycle does **not** suppress a later distinct-class liveness/hand-off nudge (it is emitted as its own message); but a second *same-class* nudge **is** suppressed (fixtures for both — distinct-class allowed, same-class collapsed).
- [ ] **Same-class-after-SENT:** a materially-new instruction of a `(PR, cycle, intent-class, worker-target)` tuple already sent this cycle is not re-sent via a new ad-hoc key; when the worker is quiescent/unreachable it rides #261/#281/#373, and when the **prior send succeeded and the worker is healthy** a fixture asserts a **distinct operator-visible escalation** fires (not a silent drop). A paired negative fixture proves a **replacement-generation target** (same PR/cycle/class) is **not** suppressed and gets its first nudge.
- [ ] **Transport-chokepoint enforcement + token binding:** the worker-send transport wrapper rejects a worker nudge lacking a valid token, and also rejects a **stale, replayed, wrong-target, or wrong-class** token (fixtures per rejection case); the token is single-use and bound to {PR, cycle key, intent-class, worker session/target, phase}; audited break-glass override path.
- [ ] **No raw-send bypass across repo-owned call sites:** a static/runtime negative control proves no repo-owned script or manual path invokes raw worker `ao send` outside the tokenized wrapper (mirrors the #318/#324 process-boundary deny of the raw verb) — every repo-owned worker-send call site routes through the gate or is the audited operator-only break-glass.
- [ ] **Legacy-record deterministic remap (no replacement suppression):** a pre-build #332 record with a **verifiable** target/generation suppresses only an equivalent current review-findings nudge for that exact target+generation; an **unverifiable** legacy record (no recoverable target) is **non-suppressing** — a fixture proves it does **not** block a replacement generation's first nudge nor a distinct intent-class, accepting at most one bounded migration-time duplicate.
- [ ] **Same-class first-claim-wins:** concurrent same-tuple candidates from two processes yield exactly one send, and the delivered content is the first-claimer's — no best-payload arbitration is asserted (each invocation carries one candidate; arbitration would need the parked collection window).
- [ ] **Closed/versioned taxonomy + incident pin:** the three incident surfaces (review-trigger, `waiting_worker_review_response`, `merge.ready`) map to the **same** review-findings intent-class (fixture); an unknown/unclassifiable nudge maps to the single versioned `unknown-worker-nudge` class whose cycle key = current PR head SHA + worker-target generation — two callers derive the same key (cross-surface fixture), it sends at most once per `(PR, head-SHA, unknown-class, target)`, a head advance arms a new one (no indefinite suppression), emits an audit reason, and does **not** mint a per-phrase bucket nor silently drop.
- [ ] **Taxonomy-version compatibility:** a previous-classifier-version review-findings record suppresses an equivalent current-version review-findings nudge for the same full tuple (fixture) — a classifier-version bump mid-cycle does not reopen duplicate sends or fail-closed-forever after restart.
- [ ] **Sibling integration shapes:** a fixture asserts the claim-around path for `ao review send` (claim acquired before invoking; not invoked when the tuple is already claimed) and the transport-token path for a pack-composed sibling send; a non-worker-observable review-channel send requires no claim and is not chokepoint-rejected.
- [ ] **Target/class scoping:** a sibling's worker-observable delivery only contends with nudges of the **same** intent-class to the **same** worker-target; a different class or a different worker-target in the same cycle is a distinct tuple and is not cross-suppressed (fixture).
- [ ] **Shell/quoting transport safety:** an LLM-composed nudge containing multiline text, quotes, backticks, `$`, `;`, markdown fences, WSL and Windows paths, and non-ASCII is delivered intact via stdin/`--file` (no shell interpolation) — fixtures per character class on the configured runtime.
- [ ] **Non-blocking dispatch:** the adapter invokes the non-blocking send (`--no-wait` or equivalent); a fixture asserts that against a busy session the call returns promptly (does not block ~600s) and the message is enqueued by AO rather than blocking the tick.
- [ ] **Legacy #332 record compatibility (verifiable vs unverifiable):** a fixture with a **verifiable**-target legacy record reads it compatibly and counts it as a prior nudge for that exact target; a separate fixture with an **unverifiable** legacy record treats it as non-suppressing (does not block a current/replacement target's first nudge) — neither path fails-closed-forever on cold start. ## Upgrade-safety check

- No AO core or vendor edits; no new AO YAML schema fields (drive via scripts + orchestratorRules prose, per #318/#332 precedent).
- No new repo secrets.
- The gate reuses the #332 shared state and the #281/#373 send wrapper — it does not introduce a second competing dedup store (a parallel store would re-open the cross-surface race this closes).
- **Journal redaction:** the gate's audit/journal records LLM-composed message context but MUST NOT persist credentials, tokens, cookies, auth headers, or raw session data; message-derived fields (session URLs, local paths, branch names) are redacted/minimized per the #281/#373 dispatch-journal redaction guarantee.
- **Backward compatibility with existing #332 records (deterministic remap, not a broad match).** The gate reads pre-existing #332 settle-state records produced by the deterministic scripts before this build — but a legacy record lacking `intent-class` and/or `worker-target` MUST NOT "broadly count as a prior nudge" (too broad → suppresses a distinct class or a replacement worker) nor be treated as unreadable (→ fail-closed forever or duplicate). It MUST be remapped by an **explicit deterministic rule** that **never suppresses a target it cannot prove it matches**: a legacy nudge record maps to the **review-findings** intent-class, and its target binding is used **only if the record's own context yields a verifiable logical target+generation**. If the legacy record's target/generation is **unverifiable** (the common case — old records lack it), it is treated as **non-suppressing for target-keyed dedup**: it does **not** block any current worker-target's first nudge (so a replacement generation always gets its first nudge), and convergence is achieved as new, properly-keyed records accrue. The deliberate trade: at most one bounded duplicate to a worker that was mid-cycle exactly at migration, never a **starved replacement**. It also never suppresses a distinct intent-class. Negative fixtures: unverifiable legacy record does not suppress a replacement-generation target nor a distinct class. Cold-start convergence over legacy state, per #111/#339.
- **Intent-class taxonomy version compatibility:** the classifier is versioned; a classifier-version bump MUST NOT orphan in-flight records — a previous-version record for an intent-class still suppresses an equivalent current-version nudge for the same full tuple (remap rule), so a deploy/restart mid-cycle does not reopen duplicate sends or fail-closed forever. This is record *semantics* compatibility, beyond the record *shape* compatibility above.

## Scenario matrix (design-analysis element 5)

Decision per orchestrator turn, per open PR: **emit a worker nudge, or suppress?** Input dimensions: SURFACE (review-trigger / review-send-followup / merge.ready-revalidate / waiting_worker_review_response ping / ci-green wake / ci-failure) × worker CYCLE STATE (still-fixing / settled-handed-off / idle-quiescent / busy-streaming) × EPISODE vs shared state (already-nudged-this-cycle / new-head-same-cycle / new-cycle) × FINDINGS (not-sent / sent+open>0 / sent+open=0) × CI (green / red / pending).

All "already nudged this cycle" cells below mean **already nudged this cycle for the same `(intent-class, worker-target)`** — a different intent-class or worker-target in the same cycle is a distinct tuple and is **not** suppressed (the distinct-class/target rows make this explicit; the tuple qualifier overrides any bare-cycle reading).

| Cycle state | Episode vs shared state | Findings | Expected |
|---|---|---|---|
| still-fixing | already nudged this cycle (same class+target) | any | SUPPRESS |
| still-fixing | new head, same cycle (same class+target) | sent, open>0 | SUPPRESS (commit = worker addressing) |
| settled/handed-off | new cycle (fresh ready head) | n/a | EMIT exactly one (single-winner) |
| idle-quiescent | already nudged this cycle (same class+target) | sent, open=0 | SUPPRESS (addressed; the incident cell) |
| any | already nudged this cycle, but DISTINCT intent-class OR DISTINCT worker-target | any | EMIT one (distinct tuple — not cross-suppressed) |
| worker busy, tuple already nudged | any | any | SUPPRESS (already served — same as any already-served tuple) |
| worker busy, tuple's FIRST nudge | any | any | DELIVER (non-blocking send; AO enqueues — no gate-side busy-defer) |
| any | already nudged this cycle by ANOTHER surface, SAME intent-class | any | SUPPRESS (cross-surface single-winner) |
| any | distinct new intent-class within the cycle, not yet sent | any | EMIT one (own claim for that class) — distinct classes are not cross-suppressed, not merged |
| any | SAME intent-class, already SENT this cycle | any | SUPPRESS — material same-class change routes to #261/#281/#373, never a new ad-hoc key |

Sibling cells at risk = **every** SURFACE row, because none of the LLM-turn surfaces currently consult the shared per-cycle state. The build must gate the shared chokepoint so all surfaces are covered — closing one surface (as #332 closed the scripts) leaves the class open on the rest.

**Surface scope (resolves the matrix-vs-out-of-scope ambiguity):** the SURFACE list above enumerates emitters; the gated command **decides** (claim/send/suppress) every **worker-observable** nudge for a tuple, whoever emits it. LLM-turn-composed nudges (review-trigger-driven, `waiting_worker_review_response`, `merge.ready`, ci-green continue-hand-off) go through it directly. `ci-failed` (#110/#342/#116) and `ao review send` keep their own logic for their **internal eligibility** (which episode/run) and any **non-worker-observable** output, but their **worker-observable** emission MUST win the tuple's claim **before** delivering — claim-before-delivery, not record-after — because a record written after the send cannot prevent a concurrent LLM send. There is no "exempt but counted" surface: owning worker-observable delivery **is** the gating.
